## 1. 项目背景与目标

### 1.1 业务背景
QRacer 服务于柔性版印刷（柔印）印前工作。印前设计师常常需要把客户提供的**二维码截图**（来自微信、抖音、网页等）转成可用于制版的矢量文件（SVG）。这个过程必须保证**像素级几何精度**——即"扫码结果一致"并不够，**码点位与原图必须 1:1 还原**。

### 1.2 工具流程
```
导入图片 → 识别码类型 → 矢量化还原 → 导出 SVG / 复制到剪贴板（粘贴进 Illustrator）
```

### 1.3 支持的码类型
| 码类型 | 几何形状 | 协议公开 | 矢量化策略 |
|---|---|---|---|
| 二维码（QR） | 方阵 | ✅ 公开（ISO/IEC 18004） | 主：解码→8 掩膜重生成；兜：网格采样 |
| 微信小程序码 | 圆形放射 | ❌ 私有 | 仅网格采样 |
| 抖音码 | 同心圆 | ❌ 私有 | 仅网格采样 |

### 1.4 核心约束
1. **几何精度优先**：码点位必须与原图 1:1，扫码一致≠合格
2. **用户可校验**：左右对比 + 差异高亮，让用户人眼确认
3. **印前工作流**：输出 SVG 文件，或把同一份 SVG 文本复制到剪贴板粘贴进 Illustrator
4. **仅 Windows**：暂不考虑跨平台
5. **轻量分发**：单 .exe 文件，目标 release < 20MB

## 2. 技术栈

### 2.1 选型表
| 维度 | 选型 | 说明 |
|---|---|---|
| 语言 | Rust 2024 edition | 性能、单二进制、内存安全 |
| GUI | `eframe` + `egui` 0.32 | immediate-mode；OpenGL 后端（`glow`）；启动时注入 Windows CJK 字体 |
| 图像 IO | `image` 0.25 | PNG/JPG/BMP/WebP |
| 图像处理 | `imageproc` 0.25 | 二值化、轮廓、形态学 |
| QR 解码 | `rxing` 0.7 | ZXing 移植；多格式 |
| QR 生成 | `qrcodegen` 1.8 | **关键**：`encode_segments_advanced` 支持 `Some(Mask::new(0..=7))` |
| 矩阵运算 | `nalgebra` 0.33 | 单应矩阵、透视变换 |
| SVG 输出 | 字符串构建 | 直接拼字符串最稳，可控 |
| 剪贴板（图像读） | `arboard` 3 | 跨平台读 |
| 粘贴/截屏 | `windows-sys` 0.60 | Windows 前台 `Ctrl+V`/`Shift+Insert` 物理按键状态检测；全屏透明遮罩框选和 GDI 屏幕捕获 |
| 剪贴板（矢量写） | `arboard` 3 | 直接写 SVG 文本，Illustrator 可粘贴为矢量 |
| 文件对话框 | `rfd` 0.15 | Native dialog |
| 错误处理 | `anyhow` 1 + `thiserror` 2 | 应用级用 `anyhow`；库层错误类型用 `thiserror` |

### 2.2 关键技术决策

#### 决策 1：immediate-mode GUI（egui）而非 retained-mode
- **背景**：备选 slint / iced / Tauri
- **选择 egui 的原因**：
  - 单一 Rust crate，无 DSL / WebView 依赖
  - 工具型 UI（按钮、对比、参数面板）非常契合 immediate-mode
  - 二进制体积小（release ~10MB）
- **代价**：每帧重画，需自己缓存重计算结果（如纹理）

#### 决策 2：小程序码/抖音码"仅几何还原，不逆向解码"
- **原因**：协议未公开，逆向后即使能还原内容，未来版本更新会失效
- **取舍**：放弃"标准化重生成"，但避免了维护私有协议解析的负担

#### 决策 3：纯 Rust 图像处理，不依赖 OpenCV
- **原因**：`opencv` crate 需本地编译 OpenCV，破坏单文件分发目标
- **代价**：`imageproc` 没有 Hough Circle，抖音码同心圆定位需自实现（轮廓 + 圆度判定）

#### 决策 4：剪贴板直接写 SVG 文本
- **原因**：手测确认 Illustrator 可直接把剪贴板中的 SVG 代码粘贴为矢量对象，且结果与导出 SVG 完全一致
- **取舍**：不再维护 EMF 生成路径，避免 EMF 尺寸、形状转换和 AI 反解 SVG 不一致的问题

## 5. 关键算法

### 5.1 QR Finder Pattern 检测
**目标**：在二值图中找到三个 "1:1:3:1:1" 嵌套方块。

```
扫描线法：
1. 对每行执行 RLE（行程编码），得到 [黑长, 白长, 黑长, 白长, 黑长]
2. 检测连续 5 段是否满足 1:1:3:1:1（容差 ±50%）
3. 对每个候选段中心做列方向同样验证
4. 满足两轴验证的候选 = finder 中心
5. 从候选中筛选模块尺寸接近、近似右角、面积较大、finder 间 timing pattern 黑白交替合理，且 7×7 finder 模板评分通过的三点组合，作为 QR 三角
```

### 5.2 透视校正
**目标**：把任意角度截图的码摆正到正方形坐标系。

```
1. 三个 finder 中心 + 推算第四个角（向量加法）
2. V2+ QR 优先搜索右下 alignment pattern；命中时用 TL/TR/BL finder 中心和 alignment 中心求模块坐标到原图坐标的单应矩阵，再反推 QR 外框四角；V1 或未命中时回退三 finder 仿射估计
3. 构造源四角 → 目标 (0,0)~(N,N) 的单应矩阵 H（nalgebra）
4. 对原彩色图做反向映射 + 双线性插值 warp 到目标图，再重新二值化；减少拍照图先二值化再拉伸造成的边缘断裂
```

### 5.3 QR 网格采样兜底
**目标**：不解码，直接在校正后图上采样每个模块。

```
1. 从校正图 finder/timing/format 区评分推断 QR 版本 → size = (version-1)*4 + 21
2. 在正式采样前，对模块网格做小范围自校准：围绕整体 shift/scale 搜索候选，用 finder、separator、timing、alignment pattern 和全模块黑白采样置信度评分，修正拍照校正后的轻微尺度/偏移残差
3. 每模块边长 = 目标图边长 / size
4. 对每个 (i, j)：
   - 取 3x3 中心子区域，使用偏向保留黑点的多数投票（9 点中 >=4 黑即判黑），降低拍照模糊导致的黑点被侵蚀
   - 写入 modules[i*size + j]
5. 可解码且已有匹配掩膜的 QR 会用原始 mask 的解码重生成矩阵作为稳定参考；若直接采样矩阵与参考差异不超过 10% 模块，则输出参考矩阵，避免中心 Logo、屏幕纹理和拍照透视导致同一码输出不一致
6. 无匹配掩膜时保留纯网格采样结果，不再用解码重生成矩阵替换；拍照样本回归会直接比较 `sample_qr_grid` 原始采样矩阵，确保该路径不依赖重生成参考
7. QR 生成阶段会保存校正图经 `sample_qr_grid` 得到的参考矩阵；差异预览和掩膜候选比较使用参考矩阵与生成矩阵逐模块比较，不再从校正图用旧采样器二次采样，避免预览假差异
7. 输出：SVG 由 size*size 个 rect 拼成；右侧预览与校正图比较差异
8. UI 中解码失败但版本可推断时，仍允许点击"网格像素匹配"，此时保留纯网格采样结果
```

### 5.4 小程序码径向采样
```
1. 三个牛眼 → 推圆心 C 和外径 R
2. 元信息区在第 4 点位置（统一坐标）→ 先解元信息
   - 元信息编码版本（36/54/72 线）
3. 已知 lines、每线 13 点：
   - 角度步进 = 360° / lines
   - 半径步进 = (R_max - R_min) / 12
4. 在每个 (line, point) 极坐标采样
5. 输出：每个点画扇形（圆心角 = 角度步进/2，半径 = r±模块半径/2）
```

阶段 5 当前实现采用几何启发式：先用黑色连通域和同心嵌套关系找三牛眼，必要时用标准左上/右上/左下象限模板匹配兜底；模板兜底会保留每个角区域的多个候选，避免灰度图中单个假候选挤掉真实牛眼。三点选择会同时检查近似等腰直角三角形、“牛眼外径 / 三角边长”比例，以及右下徽标几何位置；检测到徽标时，会优先选择推算右下角接近徽标的三牛眼组合。小程序码会先用原图三牛眼和右下徽标作为锚点校正到标准正向画布，再用目标画布上的标准三牛眼几何采样，避免旋转/轻微透视/扭曲直接进入 SVG。校正阶段先 warp 原彩色图，再对校正图重新二值化，减少二值图先 warp 带来的边缘断裂。

右下徽标检测不依赖绿色：它提取非白、非黑的大圆形/椭圆形连通域，并使用包围盒中心作为锚点，支持绿色、灰度或其它颜色的固定徽标。照片样本中浅灰背景和黑色码点会干扰连通域，因此徽标像素会同时排除过暗黑码点和浅色背景；连通域失败时，会在右下区域做圆盘形状模板扫描兜底。中心 Logo 不参与校正，因为 Logo 可能是任意形状和颜色。选中三牛眼后按近似正交轴归一化，消除单个牛眼 1-2px 的检测偏移导致的 SVG 共线误差；再通过角向相位搜索和重建误差推断 36/54/72 线版本。SVG 与预览均由同一份 `WxGrid` 径向采样结果生成，并按小程序码实际形态绘制填充圆角矩形/圆点、三牛眼定位点和右下小程序徽标。校正预览当前以 1024px 光栅化显示，右下徽标的预览 S 使用平滑曲线绘制，导出 SVG 则使用标准 SVG 子路径缩放。黑色码点粗细参考 `标准小程序码.svg`，约等于一个径向采样步长。三牛眼和徽标覆盖区会作为保留区跳过采样，避免生成原图没有的码点。

`samples/` 下 9 张标准小程序码和根目录拍照样本已纳入回归，其中 `小程序码9.png` 断言为 72 线，并限制标准样本像素差异上限；根目录 `拍照1.jpg`、`拍照2.jpg`、`拍照3.jpg` 会先校正再采样，并与 `标准.jpg` 的采样矩阵逐位对比，当前要求差异 = 0。遮挡和更低清晰度样本仍需继续补齐。

### 5.5 抖音码同心圆采样
阶段 6 当前实现不逆向抖音码 payload，只做几何还原。码类型判定使用整体证据而不是只看牛眼：`detect_kind_with_image()` 会组合 QR finder 选择、小程序码/抖音码三点选择、彩色徽标提示，以及二值图的极坐标纹理签名。这样可以避免圆形角点 QR 被误当抖音码，也避免小程序码 TL/TR/BL 三牛眼在旋转或相似形状下被误判成抖音码。

定位阶段先从黑色连通域中筛圆形/同心嵌套候选，再用三个标准角区域的 bullseye 模板扫描补漏；三点选择按近似等腰直角三角形、定位点尺寸一致性和 TL/BL/BR 方向约束评分，右上大圆徽标不参与三定位点选择。方向约束同时避免把小程序码的 TL/TR/BL 三牛眼误判为抖音码。

应用层在抖音码采样前会先使用原始三定位点把图片转正：以 TL/BL/BR 三个小同心圆和推算出的右上角为源四点，warp 到标准 TL/BL/BR 目标位置，再在转正后的彩色图和二值图上重新检测参数、徽标样式并采样。这样处理带小角度旋转的原图时，编码环相位和 SVG 标准相位处在同一坐标系内，避免直接在旋转原图上采样造成 0.x° 的整体偏移；`sample_dy*` 内部仍保留按三牛眼对角线补偿旋转的逻辑，作为测试或直接调用采样器时的保护。

采样参数由三定位点推算码圆心、定位点距离、`r_min`、`r_max` 和同心环规格。边框检测在 `r_max * 0.88..1.0` 处做 360 点径向黑度评分，同时要求 `r_max * 1.06` 的外侧黑度低于阈值；这个外侧保护用于避免无框版暗背景被误识别成黑框。无框版仍固定为 6 环、120 点/环。黑框版当前拆成 3、4 或 5 条编码环、1 个两段式外黑框和 2 条独立细环：从外往内第 1、3 个可见环是细装饰环，第 2、4、5 个可见环是固定存在的前三条编码环；第 6、7 个可见环对应两个可选内侧编码环，只在原图确有编码环时启用。细环存入 `DyDecorativeRing`，参与对外可见环数统计；外黑框存入 `DyOuterFrame`，不计入环数。细环和外黑框都不参与编码环数、72/120 点数判断和相位搜索。72/120 点数判断和相位搜索只使用前三条固定编码环，避免外框、细环和中心附近 Logo 边缘扰动整体相位。

黑框版点数在 72 和 120 两个候选中评分选择，`samples/黑框版2.jpg` 作为 120 点标准回归。版本评分仍会用前三条固定编码环搜索角向相位，但黑框版编码环最终采样和 SVG 导出都使用标准 SVG 固定相位：72 点旧布局为 `5°`，120 点旧布局为 `3°`，bullseye 徽标布局为 `2.5°`，并按三牛眼对角线方向补偿整图旋转。这样避免黑度搜索被 JPEG、抗锯齿和徽标边缘拉偏 0.x°，导致 Illustrator 重叠时三牛眼对齐但编码环整体微旋；同时采样布尔点位和最终 SVG 的点位相位保持一致。环数检测在徽标样式、右上徽标和中心 Logo 保留区确定后执行：第 4、5 个候选编码环先按 `>= 0.34` 的 3x3 投票采样，再用黑点密度、黑 run 数、平均 run 长度和最长 run 长度判断是否为真实编码环。真实编码环应是多个离散短段；中心 Logo、徽标或内侧实心区域造成的误黑通常表现为整圈黑或少数长黑弧，因此会被拒绝。这是为了支持 5/6/7 环黑框版，而不是再按点数或徽标样式硬编码环数。编码环每个 (ring, point) 以 3x3 极坐标点投票，黑框版阈值为 `>= 0.34`，无框版阈值保留 `>= 0.26`。黑框版编码环仍使用经过开闭运算清理后的二值图。黑框版编码环导出在最顶层，因此采样不再把三牛眼白底作为保留区，修复 `黑框版4.jpg` 左上牛眼附近的真实编码点漏采；无框版仍按现有绘制顺序避开三牛眼。右上徽标保留区按点数和环位区分：72 点黑框版和无框版编码环使用 `1.04 * badge.radius`；120 点黑框版最外编码环使用 `1.12 * badge.radius`，继续屏蔽紧贴徽标外圆框的误采，内侧编码环使用 `1.04 * badge.radius`，避免贴徽标但在外侧的真实编码点被误删。中心 Logo 保留区对黑框版收窄到 `0.98 * logo.radius`，因为 `黑框版11漏采点标注.png` 的最内编码环 `point53/54` 黑度均为满黑，距离中心 Logo 半径比例约 `0.983` 和 `1.005`，旧 `1.02` 或 `1.00` 扩张都会至少误删其中一个；无框版仍使用 `1.02`。`黑框版9多采点标注.png` 确认 bullseye 样式下距离比例约 `1.011` 的 ring0 point109 属于最外环徽标边缘假点，应保持空白，而距离比例约 `1.157` 的 point110 才是后续真实编码段起点；`黑框版9漏采点标注.png` 则确认 ring1 point107 黑度为满黑且距离比例约 `1.102`，应由内侧 `1.04` 规则采回。黑框版和无框版编码环只要采样中心落入各自右上徽标保留区就保持空白，不再做徽标内部强黑度补采。120 点黑框版还会额外处理最外编码环上紧邻徽标保留区、长度不超过 2 个 cell、且最近徽标距离比例在 `1.20..=1.45` 的短 run：只删除其中距离比例 `<= 1.20`、更靠近徽标内部的 cell，保留边界外侧的真实编码点。这个细分规则由 `黑框版3新问题.svg` 触发；最新标注确认红圈应包含 ring0 point96 和 point97 两个编码点，因此 point97 不再作为编码误采删除，蓝圈误采改由细环短 run 剪枝解决，并且不误删 `黑框版2.svg` 标准中真实存在的 3-cell 外层 run。后续蓝圈多采样本显示，曾为徽标外侧缺口加入的 7x5 编码环弱边缘补采会在距离徽标并不近的真实白格上被相邻黑点和抗锯齿触发，所以编码环不再做宽核补采；需要宽核保留的只限两条连续细环，因为细环是装饰线，不是离散编码格。

外黑框不再按 720 点采样成碎段，而是按 `samples/两段外黑框.svg` 的两段完整扇环生成：4 条径向切线限定两个外框段，右上徽标大圆处的绿色切线固定，另外 3 条切线在标准角附近用原图 Otsu 黑白边缘做 0.125° 级连续搜索微调；前后探针为 0.5°，避免 1.5° 探针把切线推到黑白过渡之前。两条细环仍用原图 Otsu 二值图按 720 个角向采样点独立采样，但使用细环专用采样核：每格取 5 个角向列、7 个径向偏移，同一角向列内任一径向命中即算列命中，至少 2 个角向列命中或整体黑度 `>= 0.10` 才判黑；采样时避开右上徽标区域，并做一维闭合：填补不超过 6 个采样点的小白缝、删除短于 2 个采样点的孤立黑噪。120 点黑框细环对徽标半径使用 `* 1.04` 的严格避让，保持 `黑框版2.svg` 标准覆盖；在闭合白缝前会额外删除徽标边缘距离比例 `1.04..=1.14`、长度不超过 4 个 720 角采样点的短黑 run，并修剪可被白缝闭合桥接到该短 run 的相邻尾点，避免 `黑框版3新问题.svg` 中徽标圆框外缘被当成细环输出；上限从 `1.26` 收窄到 `1.14`，因为新增根目录样图显示 `1.20+` 的外侧细环端点会被徽标边缘误剪。72 点黑框细环放宽到 `* 0.80`，让 `黑框版1.jpg` 中两条细环能贴到徽标外框，徽标内部仍由后绘制的固定 `g#b` 白底遮挡；但 `黑框版5.jpg` 这类模糊样本会把徽标圆框外缘采成可见细环残段，因此 72 点细环在闭合前后都会清理徽标边缘带，并允许边缘带外的 run 端点做 1 点弱补采。`samples/黑框版4.jpg` 的新标注显示第 2 条细环在徽标边缘清理带内存在两段被切断的真实强黑连续线，因此 72 点第 2 条细环会在清理后额外桥接不超过 10 个 720 角采样点的白 gap：只有 gap 前后都已有细环 run、gap 内每点仍是徽标边缘带且原图 5x7 核至少 5 个角向列/18 个黑采样命中时才补回。这样继续避免孤立徽标圆框残段被输出，同时恢复被清理带误切断的真实细环。定位点、右上徽标和中心 Logo 区域只作为编码环保留区跳过；外黑框遮挡关系由两段外框和后绘制的固定 `g#b` 完成。黑框版右上徽标不再依赖暗色连通域半径，因为连通域容易抓到内部 logo 而不是外侧大徽标；当前按三定位点估算徽标中心，并以小牛眼外径约 2.5 倍作为编码保留半径。无框版徽标仍使用右上象限、半径和到码圆心距离约束的连通域检测，失败时再按三定位点估算。

黑框版右上固定徽标支持旧抖音 logo 样式和从 `samples/黑框版另一种徽标样式.svg` 提取后内置的 bullseye 样式。样式选择不看颜色，因为原 logo 可能是彩色、灰度或近黑；采样时在按三定位点估算的徽标区域附近做局部形状搜索，寻找“黑中心点、白色间隔、黑同心环”的最佳签名。只有中心暗度、间隔白度和外黑环覆盖同时达标时才判为 bullseye，否则继续使用旧 logo 样式。保留区半径仍由几何估算控制，样式搜索只决定固定 SVG 徽标造型；校正预览也使用同一个 `badge_style`，旧样式画音符，bullseye 样式画同心徽标。

SVG 输出先把同一环上的连续黑采样合并成 run。黑框 72 点和 120 点布局不再 `include_str!` 或复用样本 SVG 整组内容，而是用代码内置的 viewBox、中心、徽标坐标和三牛眼坐标生成同一套固定 `g#b`；旧 logo 样式使用从新版 `samples/黑框版1.svg` 提取后内置的三色 logo path（`#fa1e5c`、`#5ffdff`、`#000`），bullseye 样式使用从 `samples/黑框版另一种徽标样式.svg` 提取后内置的右上徽标内部 path。两段外黑框先输出到 `g#a`，两条细环再按采样 run 输出到 `g#a`，随后追加对应布局的固定徽标和三牛眼造型，最后把实际采样到的 3、4 或 5 条编码环输出到 `g#c`。抖音码动态外框、细环、编码环和 bullseye 徽标仍统一使用 `#000`，旧 logo 徽标保留样本中的三色。这个绘制顺序让外黑框和细环被牛眼/徽标白底遮住，同时让编码环位于最顶层，不会被牛眼白底误遮挡；不再把外框按 72/120 数据格或 720 采样点断开。黑框版中心 Logo 不参与固定 SVG 输出，因为它是客户图案而不是码结构。无框版不使用线条描边，连续 run 用封闭填充的圆角弧条路径，单个独立点直接输出 `<circle>`，匹配无框版与小程序码类似的圆端视觉。应用流程中的差异预览按生成预览映射回转正后的二值图逐像素比较，保留区不计入差异。

`samples/黑框版*` 和 `samples/无框版*` 已纳入回归：黑框版断言为编码环 + 2 条独立细环的可见总环数，外黑框不计入，其中 `samples/黑框版1.jpg` 为 7 环（5 条编码环 + 2 条细环），`samples/黑框版2/3/4.jpg` 为 6 环（4 条编码环 + 2 条细环），`samples/黑框版6/8/9.png` 为 5 环（3 条编码环 + 2 条细环）；`黑框版2` 断言为 120 点/编码环；无框版断言为 120 点/环、6 环。`samples/黑框版2.svg` 存在时作为 120 点 fixture，回归验证旧 4 条标准编码环的采样布尔网格与标准 `g#c` 逐点比较并要求 mismatch = 0；同时解析标准 `g#a` 的细环角段，约束生成细环的 720 角覆盖和 run 数不过度偏离，并解析标准 `g#b` 前两个外框 path，要求采样外框切线与标准角度误差不超过 2°。`samples/黑框版1.svg` 存在时作为 72 点 fixture，回归验证输出不固定复用 `g#a`，固定 `g#b` 由代码内置布局生成，并确认外框为 2 段且细环 run 数不会过度碎片化；`samples/黑框版另一种徽标样式.jpg` 回归验证按形状识别为 bullseye 样式，并输出内置 bullseye 徽标 path、三牛眼和非正方形 viewBox，且黑框输出不再出现 `#231815` 或 `#221714`。另有回归确认可选内环检测会拒绝少数长弧和整圈黑的 Logo 干扰，`黑框版1.jpg` 的细环能到达徽标外框、`黑框版1漏采点标注.svg` 中靠近徽标的 72 点里层编码环能采回、`黑框版2漏采点标注.svg`/`黑框版3多采漏采点位标注.svg`/`黑框版3新问题.svg`/`黑框版4多采漏采点位标注.svg` 中红圈半径内所有编码 cell 为黑并由最终 `g#c` 编码环 run 覆盖、蓝圈不再与编码环/细环/外框动态输出重叠、根目录 `黑框版4.jpg` 的左上牛眼附近编码点不会被牛眼白底保留区误删、`黑框版9多采点标注.png` 的 bullseye 徽标下沿最外环假编码点保持空白、`黑框版9漏采点标注.png` 的内侧 ring1 point107 能采回、`黑框版11漏采点标注.png` 的最内编码环真实点不会被中心 Logo 保留区误删、徽标上沿 120 点外细环端点不会被误剪、左下方向 `4/9` 但后接长黑段的编码起点能保留，并确认所有黑框版/无框版编码环在右上徽标保留区内保持空白。新增透视回归验证带小角度旋转的抖音三定位点会先 warp 到标准 TL/BL/BR 目标位置；应用层回归直接处理 `samples/黑框版4.jpg`，确认完整流程保留转正二值图且输出网格对角线回到 45°。标准 `g#c` 解析按 path 包围盒中心映射到 4x120 网格，避免 Illustrator 路径从不同边界起笔时把同一个码点错映射到相邻 cell。回归同时验证三同心圆定位、参数检测、徽标检测、采样黑点数下限，以及 SVG 不会退化成只有徽标的输出。

```
3. 原图黑、生成图白的模块染红；原图白、生成图黑的模块染蓝
4. QR 原掩膜显示不是只看 format info；8 种掩膜中必须存在一个在中心 Logo 区外差异为 0 的矩阵才算匹配。中心 Logo 区内的少量差异不影响匹配；只要中心外有差异，就显示"无匹配掩膜"，不显示"原掩膜 x"，并自动切到网格像素匹配采样
5. 小程序码使用 `wx_grid_to_diff_preview_image` 做像素级对比，红/蓝含义与 QR 一致，徽标和中心 Logo 区域忽略
6. 掩膜面板/小程序码面板都提供"显示差异"开关；状态栏在差异数后写明红/蓝含义
7. 原图坐标系半透明叠加需要保留 homography 后再做反投影，可作为后续增强
```

## 6. 错误处理

### 6.1 分层策略
- **库层（detect / pipeline / codec / vector / clipboard）**：返回 `Result<T, QRacerError>`（用 `thiserror`）
- **应用层（app / ui）**：用 `anyhow::Result`，错误转字符串写入 `status` 字段
- **panic 哲学**：只在真正违反不变式时 panic（如 finder 数组长度不为 3），其他都返回 Err

### 6.2 关键错误类型（阶段 2 已引入）
```rust
#[derive(thiserror::Error, Debug)]
pub enum QRacerError {
    #[error("找不到三个定位点（找到 {0} 个）")]
    InsufficientFinders(usize),
    #[error("无法判定码类型")]
    UnknownCodeKind,
    #[error("QR 解码失败：{0}")]
    QrDecode(String),
    #[error("剪贴板访问失败：{0}")]
    Clipboard(String),
    #[error("IO 错误：{0}")]
    Io(#[from] std::io::Error),
}
```

## 7. 并发模型

### 7.1 当前后台处理
粘贴、打开或截屏得到 `DynamicImage` 后，UI 线程立即写入 `original` 并清空旧预览，状态栏显示 spinner + 进度条。预处理、码类型识别、透视校正、解码/采样、SVG 和预览生成放到后台线程执行，通过 `std::sync::mpsc` 把 `ProcessResult` 发回 UI。

UI 线程每帧 `try_recv()` 拉取结果，不阻塞；后台任务未完成时持续 `request_repaint_after()`，保证 loading 状态刷新。用户再次导入图片时会替换当前 receiver，旧任务结果自然丢弃。

### 7.2 截屏框选
点击"截屏"后先向主窗口发送 `ViewportCommand::Minimized(true)`，后台线程短暂等待窗口最小化，再创建 Win32 顶层透明遮罩窗口。用户拖拽框选区域后，遮罩销毁，使用 GDI `BitBlt` 捕获所选屏幕矩形，并按普通导入流程进入后台处理；Esc 或右键取消时恢复主窗口并写入状态栏。截屏线程结束时会主动 `request_repaint()`，排队 `ViewportCommand::Minimized(false)`，并用 Win32 `FindWindowW("QRacer")` + `ShowWindow(SW_RESTORE)` 做原生还原兜底，确保任务栏入口保留且窗口可恢复。

### 7.3 后续抽象
当前只有图片导入处理和截屏两个后台任务，逻辑直接收敛在 `QRacerApp` 内。SVG 文本写剪贴板耗时短，仍保持同步调用；若阶段 6/7 引入批量处理或更长耗时任务，再抽出独立 `job::runner`。
